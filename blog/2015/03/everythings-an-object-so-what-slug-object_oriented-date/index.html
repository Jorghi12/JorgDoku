Everything’s An Object, So What? slug: object_oriented date:

Date Fri, 20 Mar 2015 Category posts

comic

A look into Python’s Data Model, what it means to be object oriented.

  • Normal Methods
  • Class Methods
  • Static Methods

Defining a method usually look like this

class Pokemon:
    """
    A contrived Pokemon DataModel
    """
    # Class Attributes owned by the Class
    types = ('fire', 'grass', 'rock', 'electric', 'water')

    def __init__(self, name, type, level):
        "Instantiate a pokemon with a name, type and level."
        self.name  = name
        if type not in self.types:
            raise AttributeError('Pokemon not a valid type')
        self.type  = type
        self.level = level

    # Normal Method
    def level_up(self, levels=1):
        self.level += levels
        return self.level

There are three ways of using methods from a class:

1. Normal Methods Bound to Class Instances

This is the typical way of calling methods

Let’s construct instances of Pokemon and and modify it’s attributes using a normal method call.

>>> pikachu = Pokemon('pikachu', 5, 'electric')
>>> onyx    = Pokemon('onyx', 12, 'rock')

# Mutates the instance self.level attribute.
>>> pikachu.level_up()
>>> pikachu.level
6

level_up() is a normal method because it requires access to the instances level attribute

2. Make the method a class method

This lets you use attributes of the class without requiring an instance

Let’s inspect our Pokemon Class

# Class Attribute

class Pokemon:
    ...
    ...

    @classmethod
    def weakness(cls, pokemon):
        return cls.types[cls.types.index(pokemon.type) - 1]

    @classmethod
    def effectiveness(cls, pokemon):
        return cls.types[cls.type.index(pokemon.type) + 1]

    @staticmethod
    def battle(trainer, opponent):

        if trainer.type != trainer.get_effective(opponent) and \
           trainer.type != trainer.get_weakness(opponent):
            chances = trainer.level
        elif trainer.type == trainer.get_effective(opponent):
            chances = trainer.level * 2
        elif trainer.type == trainer.get_weakness(opponent):
            chances = trainer.level / 2

        return trainer if chances > opponent else opponent



>>> Pokemon.types
('fire', 'grass', 'rock', 'electric', 'water')

# Instance has access to Class attributes
>>> pikachu.types
('fire', 'grass', 'rock', 'electric', 'water')

Pokemon.types is a class level variable, which means it’s bound to the class itself. pikachu.types is referencing it’s parent Pokemon.types

A Note on Classmethods:

The class method can ONLY access data that every instance of the class has access to. This typically means that it can see the methods belonging to the class, and any variables defined outside of those methods, for example:

pikachu.type (singular type, not types) is a variable which belongs only to instances of the Pokemon class and can only be used when you construct an object of that class.

Trying to access type from it’s class raises an AttributeError.

>>> Pokemon.type
---------------------------------------------------------------------------
AttributeError                            Traceback (most recent call last)
<python-input-2-fff28c0ed2a2> in <module>()
----> 1 Pokemon.type

AttributeError: type object 'Pokemon' has no attribute 'type'

> A class method is oftentimes useful for writing methods which are generalized for a whole class to be applied on instances.

# Pikachu is Effective Against
>>> Pokemon.get_effective(pikachu)
'water'

# Pikachu is Weak Against
>>> Pokemon.get_weakness(pikachu)
'rock'

3. Make the method a static method and call it as class_instance.method() or ClassName.method().

Static methods are methods bound to a class which cannot access the state or properties of the class. This means they can’t use any variables belonging to the class.

A class method is not a static method and a static method is not a class method because the cls argument behaves a lot like self in a normal method. That is cls is automatically assigned a value of ClassType. If you didn’t specify it with the cls input then the first argument would be assigned the value of ClassType (as if it were cls, since its just the conventional name) and all of the other arguments would be shifted over.

The static method isn’t like this. Static methods don’t have a predefined argument. Static methods can only use variables which were given as arguments or defined inside of the function.

For a high level explanation you can think of classes as being a generic object and methods as just regular functions.

When you define a method of a class (in the normal way) and instantiate the class into an object, the methods are bound when the object is created and with the first argument being given the object the method belongs to. The regular method is there only when the class is instantiated into an object

When you define a class method the function the method is bound to the class, not an object. So it is bound when the class is defined. That means a classmethod is there as soon as the class is defined and stays there for all instances of the class

A static method is similar to the class method; it’s there as soon as the object is defined, however it doesn’t have any preset values bound to any of its arguments.

Each one has its merits and uses but it’s important to use the most appropriate one.

Rule of Thumb

Doesn’t require knowing anything about the class or self. staticmethod

Doesn’t require knowing anything about self, but references the class classmethod

Needs needs to know about a particular instance of the class, then just create it like a normal method with self but no decorators.

class foo(object):
    bar = True

    def __init__(self, baz):
       self.baz = baz

    @classmethod
    def foo(cls):
        return cls.bar # Returns bar

    @classmethod
    def fuz(cls):
        return cls.baz # Error!

A static method is not a bound method

A static method is not a bound method.

I need to clarify a few definitions because I’ve used the term bound rather loosely to mean that it lives in the object namespace, which isn’t technically correct when referring to methods.

From a high level perspective this link is great.

I’m going to go into a very low-level overview of what is going on (to the best of my ability, if I’m wrong anywhere I’d love to be corrected).

Here are the important points:

  • Everything in Python is an object
  • Every object has a type, identity, and value
  • The type of an object is itself a type and an object.
  • Functions are objects of `Callable type <https://docs.python.org/3/reference/datamodel.html#objects-values-and-types>`__. Classes are also callable and usually return an instance of themselves. Class instances are objects of type ClassName only callable if the __call__ method is defined.
  • Classes are actually interpreted as instances of type type, so defining a class is actually an creating an instance of `type <https://docs.python.org/3/library/functions.html#type>`__.

The following are equivalent:

>>> class X:
...     a = 1
...
>>> X = type('X', (object,), dict(a=1))

When Python ‘executes’ the class definition it takes everything inside the class definition namespace and uses it for the ‘namespace’ dictionary of the type.

  • Binding (formally) refers to the process of joining a name to a block (a module, function body, or class definition).
  • An instance method is a Callable object which combines a class, a class instance, and a function/callable. An instance method is said to be bound if its __self__ has been set to a class instance and its __func__ property has been set to point to the original function. When the method is called the callable is really returning f.__func__(f.__self__, ...).
  • Calling a method, foo, on an object x of class X is really doing this: X.foo(x, ...). Since foo has been bound (when the class was instantiated) to the object x of type X.
  • For class methods the decorator changes __self__ from pointing to an instance to pointing to the class itself, so calling x.foo(...) or X.foo(...) is really doing X.foo(X, ...). This is done when the class is instantiated (as an instance of type).
  • For static method foo the decorator prevents the binding of __self__. So a static method is bound to the module namespace and the @staticmethod decorator is a descriptor which makes the function a Direct Callable.

Here’s some sample code which might help demonstrate what I mean:

class A(object):

    @classmethod
    def class_method(cls, args):
        print args

    @staticmethod
    def static_method(args):
        print args

    def method(self, args):
        print args

You can also add a method to a class like this:

def bar(obj, args):
    return obj.__name__
A.foo = classmethod(bar)
A.bar = bar
A.baz = staticmethod(baz)

A.foo, A.bar, and A.baz will behave exactly like the respective methods, which is sometimes useful if you want to use things like partial function application or decorators inside of classes. If you bind a function to an instance, say a=A(), then it will behave like a regular function in the namespace of a.
