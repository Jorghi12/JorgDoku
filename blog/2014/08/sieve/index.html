Prime Sieves and Python Data Structures

Date Fri, 29 Aug 2014 Tags Python

Recently, I’ve been working through Project Euler problems in order to improve my core programming skills.

One of those recurring problems requires an efficiently calculating and testing for prime numbers. The first algorithm that comes to mind is The Sieve of Eratosthenes. The Sieve, is one of many prime sieves, and is a simple yet time efficient algorithm for finding all the primes below a certain limit. It can be shown to have a time complexity of O(nloglogn).

The Algorithm

  1. Make a table one entry for every number 2 ≤ n ≤ limit
  2. Starting at 2, cross out all multiples of 2, not counting 2 itself.
  3. Move up to the next number that hasn’t been crossed out
  4. Repeat Step 2-3 up till √(n)

Visually we can depict each loop removing values from the list of real numbers until all that is left are the primes.

Unfortunately, this solution starts to become less viable for larger problem sizes. Since it requires a table of every number to the last integer in memory, the space complexity of sieves generally grows in the order of O(n). In order to deal with memory issues there is a set of Sieve algorithms called segmented sieves which distribute the problem into smaller subsets.

Here’s how it would look in Python.

Sieve

def sieve(n: int) -> list:
    """Sieve away and only primes are left."""
    primes = 2*[False] + (n-1)*[True]
    for i in range(2, (n**0.5+1)):
        for j in range(i*i, n+1, i):
            primes[j] = False
    return [prime for prime, checked in enumerate(primes) if checked]

It’s important to notice we implement lists as the main data structure. Remember although the algorithm requires an in memory table containing, we have the freedom to construct it using whatever tools the runtime affords. Here is an example of another sieve but with it’s guts swapped with unordered sets.

Sets: Unordered Collections

def set_sieve(n: int) -> set:
    """
    Sets are mutable, unordered, hash collections which are useful
    for quick membership testing.
    """
    primes = set(range(2, n+1))
    for i in range(2,int(n**0.5+1)):
        if i in primes:
            primes -= set(range(i*i, n+1, i))
    return primes

The primary difference you see with using sets and lists, is the lack of a list-comprehension for composing the function return value. Also the second for-loop substituted with the binary -=, assignment operator, which for set objects has been overloaded with a difference update expression or the set mathematical compliment ∀{x|x ∈ A ⊂ B}. Sets are a great way represent mathematical operations and carry out membership testing on unordered groups of objects.

Utilizing sets for mathematical operations allows the code to be easier to read and understand. It also provides us a mechanism of understanding algorithms in cleaner syntax. For instance, in the previous example, we are first creating a set of all values, and iteriatively deleting items from that set. It would also be correct to create a set containing the factors, and then perform insertion over all the factors into it. This takes advantage of the fact that set insertions perform better than set deletions due to not needing to perform a traversal.

Insertion Sets:

def set_insertion_sieve(n: int) -> set:
    """
    Performing insertion over deletion utilizing
    constant time nature of hashable collections.
    """
    factors = set()
    for i in range(2,int(n**0.5+1)):
        if i not in factors:
            factors |= set(range(i*i, n+1, i))
    return set(range(2,n+1)) - factors

While fundamentally the choice data structure is orthoganal to program correctness, it’s important for real runtime performance to use the best tool to fit the job. Lists and sets are great general purpose data-structure, and is likely the first tool to reach for in the arsenal. It’s hard not to overuse or abuse them, when other more viable data-structures exists. Maslow tells us "If the only tool you have is a hammer everything looks like a nail". In Python, the generality of lists and sets are probably also their downside. We want to use them for everything.

So what is the optimal data structure to perform this calculation?

Numpy

The N-Dimensional Array
import numpy as np

def np_sieve(n: int) -> np.ndarray:
    """
    Sieve with it's guts swapped with numpy
    ndarray
    """
    primes = np.ones(n+1, dtype=np.bool)
    for i in np.arange(2, n**0.5+1, dtype=np.uint32):
        if primes[i]:
            primes[i*i::i] = False
    return np.nonzero(primes)[0][2:]

Numpy is a third party package containing fast array based data structures and methods for utilizing them. Here we are utilizing numpy’s n-dimmensional array, which are continguous strides of memory with fixed width statically typed elements. In our example we are filling up the array with an 8-bit bool. The downside of numpy arrays is that they require that each member be of the same data type, unlike lists which can be dynamically allocated to fit any data type. The benefit is that we now have random access data-structure that can be accessed based on it’s position in memory, without requiring any traversal.

Testing for Primality

So now that we’ve written the sieve in a bunch of different ways. How do we know that each way is indeed correct.

The most obvious way to figure if a number’s prime, is to try dividing the number by all the numbers between 2 ≤ n ≤ n − 1. One optimization would be to only factor these numbers up to 2 ≤ n ≤ √(n). This is due to the fact that if n = p × q, either p or q must be less than or equal to √(n), or their product is greater than n. If you check possible factors up to √(n), you will find the smaller factors, and when you divide n by that factor, you will find the other one. This reduces the run time to O(√(n)).

def all_primes(primes: iter) -> bool:
    for prime in primes:
        if any(prime % n == 0 for n in range(2, int(prime**0.5))):
            return False
    return True
Now lets test to see if our sieves can correctly find the first 100 prime numbers
>>> all_primes(sieve(10**3))
True

>>> all_primes(np_sieve(10**3))
True

>>> all_primes(set_sieve(10**3))
True

>>> all_primes(set_insertion_sieve(10**3))
True

A Timer

Now that these sieves are indeed correct, lets see how fast they run. We will construct a timer taking advantage of what is known as a context manager. Basically, a context manager allows us to use the with construct and will perform an operation before and overloading the __enter__ and __exit__ methods of python object. We can create a context manager merely by using a decorator from the contextlib module.

import time
from contextlib import contextmanager

@contextmanager
def timer(label):
    start = time.time()
    try:
        yield
    finally:
        end = time.time()
    print('{label}: {time:03.3f} sec'.format(
        label=label, time=end-start)
    )
It’s also possible to use Python’s built in timeit module, which can deal with much more complex and isolated timing instances.
